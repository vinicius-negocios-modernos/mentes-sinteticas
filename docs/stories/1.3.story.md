## Descricao
Como desenvolvedor do Mentes Sinteticas, quero que as variaveis de ambiente e configuracoes sensveis sejam gerenciadas de forma segura e validadas na inicializacao, para que segredos nao vazem e o app falhe cedo com mensagens claras quando mal configurado.

## Contexto
O prototipo tem multiplos problemas de seguranca relacionados a configuracao. A Gemini API key esta em `.env.local` plaintext sem nenhuma validacao. O script `list_models.ts` expoe a key em query params (visivel em logs de rede). O modelo (`gemini-2.0-flash`) e parametros de geracao (temperatura, topK, topP, maxOutputTokens) estao hardcoded em `gemini.ts`. Nao existe um schema de validacao para env vars -- se alguma estiver ausente, o sistema pode se comportar de forma imprevisivel ou crashar.

Esta story prepara a fundacao para a Fase 2 (Supabase), que adicionara mais env vars (`SUPABASE_URL`, `SUPABASE_ANON_KEY`, etc).

## Debitos Resolvidos
- **SYS-005**: API key exposure risk -- `list_models.ts` expoe key em query params (ALTO, 2-3h)
- **SYS-022**: Hardcoded model name e generation config -- `gemini-2.0-flash`, temperatura, topK, topP hardcoded (MEDIO, 2-3h)

## Notas Tecnicas
- **Zod para env vars:** Pattern recomendado para Next.js:
  ```typescript
  import { z } from 'zod';

  const envSchema = z.object({
    GEMINI_API_KEY: z.string().min(1, 'GEMINI_API_KEY is required'),
    GEMINI_MODEL: z.string().default('gemini-2.0-flash'),
    GEMINI_TEMPERATURE: z.coerce.number().min(0).max(2).default(0.7),
    // ...
  });

  export const config = envSchema.parse(process.env);
  ```
- **Valores atuais hardcoded em gemini.ts (~linha 55-60):**
  - model: `"gemini-2.0-flash"`
  - temperature: nao explicito (default do SDK)
  - topK: possivelmente configurado
  - topP: possivelmente configurado
  - maxOutputTokens: `8192`
- **list_models.ts:** Este script e usado para debug/desenvolvimento. Verificar se usa o SDK diretamente ou faz chamadas REST manuais (que podem expor a key na URL).
- **Preparacao para Fase 2:** Na Fase 2, `SUPABASE_URL` e `SUPABASE_ANON_KEY` serao adicionados ao schema. O design deve ser extensivel -- adicionar novas env vars deve ser trivial.
