# Story 2.2: Autenticacao com Supabase Auth + Middleware
**Epic:** 1 -- Transformacao Mentes Sinteticas
**Fase:** 2 -- Fundacao (DB, Auth, CI/CD)
**Status:** Done
**Estimativa:** 5 SP (1 SP = ~2h)
**Criado por:** @sm (River)

## Descricao
Como usuario do Mentes Sinteticas, quero me autenticar com email/senha para que minhas conversas sejam privadas e persistentes, e para que rotas protegidas (/chat/*) sejam acessiveis apenas por usuarios logados.

## Contexto
O sistema atualmente tem ZERO autenticacao -- qualquer pessoa que acesse a URL consegue usar o chat e consumir a API key do Gemini. Isso e o debito mais critico (SYS-001). Esta story implementa Supabase Auth com email/password, middleware de protecao de rotas, e paginas de login/signup. A Story 2.1 (Supabase + Drizzle) ja esta DONE -- o schema inclui `conversations.user_id` referenciando `auth.users`, e RLS policies ja existem em `supabase/rls.sql` aguardando auth ativo.

Stack: Supabase Auth via `@supabase/ssr`, Next.js 16.1.1 middleware, server components/actions.

## Debitos Resolvidos
- **SYS-001** (CRITICO): Zero authentication -- implementacao completa de auth
- **SYS-027** (ALTO): No middleware -- middleware de protecao de rotas
- **CROSS-002** (parcial): Auth component do sistema de autenticacao

## Notas Tecnicas
- `@supabase/ssr` e o pacote recomendado para Next.js App Router (substitui `@supabase/auth-helpers-nextjs` deprecated).
- O `createClient` de server precisa receber cookies via `cookies()` do Next.js para manter sessao server-side.
- O `createBrowserClient` usa `NEXT_PUBLIC_SUPABASE_URL` e `NEXT_PUBLIC_SUPABASE_ANON_KEY` (ja no `.env.example` da Story 2.1).
- middleware.ts deve usar `NextResponse` e `updateSession` pattern do Supabase SSR para refresh automatico do token.
- Matcher do middleware: `/((?!_next/static|_next/image|favicon.ico|auth/callback).*)` -- proteger tudo exceto assets e callback.
- Paginas de login/signup devem ser server components com client components embarcados para formularios interativos.
- RLS policies de `supabase/rls.sql` (Story 2.1) passam a ser efetivas com auth ativo -- `auth.uid()` retorna o user logado.
- Para esta story, apenas email/password. OAuth providers (Google, GitHub) serao adicionados em story futura.
