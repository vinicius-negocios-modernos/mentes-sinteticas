# Story 2.3: Persistencia de Conversas (CRUD + RLS)
**Epic:** 1 -- Transformacao Mentes Sinteticas
**Fase:** 2 -- Fundacao (DB, Auth, CI/CD)
**Status:** Done
**Estimativa:** 5 SP (1 SP = ~2h)
**Criado por:** @sm (River)

## Descricao
Como usuario autenticado do Mentes Sinteticas, quero que minhas conversas sejam salvas no banco de dados para que eu possa retomar conversas anteriores sem perder historico, e para que cada usuario tenha acesso apenas as suas proprias conversas via RLS.

## Contexto
Atualmente o chat e efemero -- todas as mensagens sao perdidas ao recarregar a pagina (debito UX-004 CRITICO). Com a Story 2.1 (schema DB) e Story 2.2 (auth) completas, temos schema pronto (`conversations`, `messages`) e `user_id` disponivel via sessao. As RLS policies existem em `supabase/rls.sql` mas ainda nao foram aplicadas. Esta story implementa o CRUD completo de conversas e mensagens, integra com o fluxo de chat existente, e ativa RLS para isolamento por usuario.

## Debitos Resolvidos
- **UX-004** (CRITICO): Chat history lost on refresh -- persistencia completa
- **CROSS-002** (parcial): Persistence layer para conversas
- **CROSS-003** (parcial): Arquitetura stateful com dados persistidos

## Notas Tecnicas
- Schema ja existe: `conversations` (id, user_id, mind_id, title, created_at, updated_at) e `messages` (id, conversation_id, role, content, token_count, created_at).
- RLS policies em `supabase/rls.sql` -- aplicar via migration Drizzle ou Supabase dashboard.
- `user_id` disponivel via `supabase.auth.getUser()` no server (Story 2.2).
- Servicos devem usar Drizzle ORM (configurado na Story 2.1) para queries tipadas.
- Titulo auto-gerado: truncar primeira mensagem do usuario em ~60 chars.
- Cascade de delete: configurado no schema FK (`onDelete: 'cascade'`), mas verificar se ativo.
- Lista de conversas pode ser server component com fetch no load da pagina.
- Ao retomar conversa, carregar mensagens ordenadas por `created_at` ASC.
