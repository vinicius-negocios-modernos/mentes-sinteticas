# Story 2.7: Security Headers + Health Check + Prompt Security Hardening
**Epic:** 1 -- Transformacao Mentes Sinteticas
**Fase:** 2 -- Fundacao (DB, Auth, CI/CD)
**Status:** InProgress
**Estimativa:** 5 SP (1 SP = ~2h)
**Criado por:** @sm (River)

## Descricao
Como administrador do Mentes Sinteticas, quero que a aplicacao tenha security headers configurados (CSP, HSTS, X-Frame-Options etc.), um endpoint de health check estruturado (/api/health) e hardening de seguranca nos prompts enviados ao Gemini API — para proteger contra ataques comuns (clickjacking, MIME sniffing, prompt injection) e permitir monitoramento de saude do sistema em producao.

## Contexto
A aplicacao nao possui security headers HTTP, o que a deixa vulneravel a clickjacking, MIME sniffing e outros ataques client-side. Nao existe um endpoint de health check para monitoramento — essencial para deploys em producao (Vercel, uptime monitors). Alem disso, o system prompt enviado ao Gemini API em `src/lib/gemini.ts` nao possui instrucoes defensivas contra prompt injection — um usuario pode tentar manipular o modelo para sair do personagem ou revelar instrucoes internas. Esta e a ULTIMA story da Fase 2 — apos sua conclusao, a fundacao estara completa para iniciar a Fase 3 (Core UX).

## Debitos Resolvidos
- **SYS-005**: No security headers — aplicacao sem CSP, X-Frame-Options, HSTS etc.
- **SYS-006**: No health check endpoint — sem monitoramento de saude do sistema.
- **SYS-007**: No prompt security hardening — system prompt vulneravel a injection/jailbreak.

## Notas Tecnicas
- Next.js 16 suporta `headers()` async em `next.config.ts` para definir headers customizados por rota. Usar pattern `/:path*` para aplicar a todas as rotas.
- CSP deve usar `'self'` para scripts, `'self' 'unsafe-inline'` para styles (necessario para Tailwind CSS inline styles), e `connect-src` deve incluir dominios do Supabase (`*.supabase.co`) e Gemini API (`generativelanguage.googleapis.com`).
- Health check nao deve requerer autenticacao — e um endpoint publico para monitoramento externo.
- Para verificar DB: usar `db.execute(sql\`SELECT 1\`)` via Drizzle. Wrap em try/catch com timeout de 5s.
- Para verificar Auth: usar `supabase.auth.getSession()` ou similar — o objetivo e confirmar que o servico Supabase Auth responde.
- Prompt security: adicionar ao final do system prompt existente, NAO substituir. Manter tom natural que se integre ao personagem da mente.
- `package.json` version pode ser importado diretamente em API routes para o campo `version` do health check.
- HSTS so deve ser habilitado em producao (HTTPS). Em dev (localhost HTTP), HSTS causa problemas. Considerar condicional via `NODE_ENV`.
