# Story 3.4: Error Boundaries + Error Handling Strategy
**Epic:** 1 -- Transformacao Mentes Sinteticas
**Fase:** 3 -- Core UX + Chat Experience
**Status:** Done
**Estimativa:** 5 SP (1 SP = ~2h)
**Criado por:** @sm (River)

## Descricao
Como usuario do Mentes Sinteticas, quero que erros durante o uso do app sejam tratados de forma elegante e informativa â€” sem telas brancas, sem mensagens genericas, sem perda de contexto â€” para que minha experiencia seja confiavel e eu saiba exatamente o que aconteceu e o que posso fazer. Como desenvolvedor, quero uma estrategia de error handling padronizada que defina como erros fluem do servidor ate a UI, quais tipos geram toasts vs inline errors, e que loading states (Skeleton) cubram todas as areas assincronas.

## Contexto
O app ja possui dois `error.tsx` do Next.js (root e chat/[mindId]) com UI basica inline (Story 1.2), mas nao ha React Error Boundaries em componentes individuais, nao ha loading states com Skeleton, e nao ha estrategia padronizada de como erros sao exibidos ao usuario (toast vs inline vs fallback). O Sonner (toast) esta instalado e configurado (Story 3.1), e o componente Skeleton do shadcn/ui ja existe em `src/components/ui/skeleton.tsx`. A API Route de streaming (`src/app/api/chat/route.ts`) ja retorna erros estruturados, mas o client-side nao os trata de forma consistente.

## Debitos Resolvidos
- **UX-003**: No graceful error handling â€” erros causam tela branca ou mensagens cruas
- **UX-004**: No loading indicators â€” usuario nao sabe se algo esta carregando
- **ARCH-005**: Inconsistent error propagation â€” cada area trata erros de forma ad-hoc

## Notas Tecnicas
- **React Error Boundaries:** Devem ser class components (React nao suporta error boundaries com hooks). O componente wrapa `componentDidCatch` e `getDerivedStateFromError`. Para Next.js 16+, os `error.tsx` ja existentes cobrem erros de rota â€” os ErrorBoundary desta story cobrem erros dentro de componentes individuais (granularidade mais fina).
- **Sonner:** Ja instalado e renderizado no root layout via `<Toaster />`. API: `toast.error("msg")`, `toast.warning("msg")`, `toast("msg", { action: { label, onClick } })`. Suporta duration, description, e action buttons.
- **Skeleton:** O componente `src/components/ui/skeleton.tsx` do shadcn/ui renderiza um `<div>` com animacao pulse. Compor multiplos Skeletons para simular o layout real de cada area.
- **classifyError heuristica:** HTTP 401/403 -> AuthError, HTTP 429 -> RateLimitError, fetch TypeError/AbortError -> NetworkError, ReadableStream errors -> StreamError, Zod errors -> ValidationError, fallback -> AppError generico.
- **Nao inclui:** Retry automatico com backoff (Fase 4), offline detection (Fase 4), Sentry integration (Story 4.4), Suspense boundaries do React (Story 4.7).
