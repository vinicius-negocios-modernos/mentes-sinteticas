# Story 3.6: Type Safety + Model Config Externalization
**Epic:** 1 -- Transformacao Mentes Sinteticas
**Fase:** 3 -- Core UX + Chat Experience
**Status:** InProgress
**Estimativa:** 5 SP (1 SP = ~2h)
**Criado por:** @sm (River)

## Descricao
Como desenvolvedor do Mentes Sinteticas, quero eliminar todas as type assertions inseguras (`as X`) e dados externos nao-validados no codebase, e externalizar a configuracao do modelo AI em um modulo dedicado com suporte a presets тАФ para que o sistema seja type-safe de ponta a ponta, a troca de modelo seja trivial (sem editar codigo), e o runtime rejeite configuracoes invalidas antes de chegar ao Gemini.

## Contexto
O codebase ja usa `strict: true` no `tsconfig.json` e nao possui `any` tipos explicitos. Porem existem **type assertions inseguras** em varios pontos:
- `JSON.parse(data) as Manifest` em `src/lib/ai/knowledge.ts` тАФ sem validacao runtime
- `fallbackResponse.text as string` em `src/components/chat/chat-interface.tsx` тАФ narrowing bypass
- `formData.get("email") as string` em `src/app/login/page.tsx` e `src/app/signup/page.tsx` тАФ FormData pode retornar `null`
- `as HTMLDivElement | null` em `src/components/chat/chat-interface.tsx` тАФ DOM query sem type guard
- `theme as ToasterProps["theme"]` em `src/components/ui/sonner.tsx` тАФ narrowing bypass

Para o modelo AI: `src/lib/config.ts` ja valida env vars com Zod (`GEMINI_MODEL`, `GEMINI_TEMPERATURE`, etc.) e ambos `chat.ts` e `stream.ts` consomem via `getConfig()`. O proximo passo e criar um modulo dedicado `src/lib/ai/config.ts` que centraliza a configuracao AI, oferece presets nomeados (ex: `creative`, `precise`, `balanced`), e facilita a troca de modelo sem editar multiplos arquivos.

## Debitos Resolvidos
- **SYS-018**: Unsafe type assertions (`as X`) e dados externos nao-validados тАФ `JSON.parse` sem Zod, `formData.get` sem null-check, DOM queries sem type guard
- **SYS-022**: Configuracao do modelo AI dispersa тАФ embora ja externalizada para env vars, nao ha presets nomeados, nem modulo dedicado que centralize a logica de configuracao AI separada da config geral da app

## Notas Tecnicas
- **Manifest validation:** `readManifest()` ja retorna `null` em caso de erro. Adicionar `safeParse` mantem o mesmo contrato mas com validacao estrutural. Log `console.warn` em caso de schema mismatch para debugging.
- **Presets override logic:** Env vars individuais (`GEMINI_TEMPERATURE`, etc.) tem prioridade sobre o preset. Se `AI_PRESET=creative` e `GEMINI_TEMPERATURE=0.5`, a temperatura sera 0.5 (env var explicita vence).
- **Backward compatibility:** Se nenhum `AI_PRESET` for definido e nenhum env var individual for setado, os defaults atuais sao mantidos (model: `gemini-2.0-flash`, temp: 0.7, topK: 40, topP: 0.95, maxTokens: 8192) тАФ zero breaking changes.
- **FormData null-check:** `formData.get()` retorna `FormDataEntryValue | null`. A abordagem correta e `const email = formData.get("email"); if (!email || typeof email !== "string") return;`.
- **Nao inclui:** Multi-provider support (ex: OpenAI + Gemini), runtime model switching via UI (Fase futura), per-mind model config (Fase futura).
