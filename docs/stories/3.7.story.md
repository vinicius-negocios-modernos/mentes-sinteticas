# Story 3.7: Context Window Optimization
**Epic:** 1 -- Transformacao Mentes Sinteticas
**Fase:** 3 -- Core UX + Chat Experience
**Status:** InProgress
**Estimativa:** 7 SP (1 SP = ~2h)
**Criado por:** @sm (River)

## Descricao
Como usuario do Mentes Sinteticas, quero que o sistema gerencie de forma inteligente o contexto enviado a cada requisicao ao Gemini — limitando o historico de mensagens, enviando apenas os file URIs da mind ativa, e estimando tokens para evitar estourar o context window — para que as conversas longas continuem funcionando sem degradacao, o custo de tokens seja reduzido, e a latencia de resposta diminua.

## Contexto
Atualmente, duas ineficiencias criticas existem no pipeline de chat:

1. **Knowledge injection sem filtro (SYS-012):** A funcao `getFileParts(mindName)` ja filtra por mind via DB, mas a funcao `buildStreamMessages()` em `stream.ts` (e `createMindChat()` em `chat.ts`) injeta TODOS os file URIs da mind em TODA requisicao. Cada mind pode ter 7+ arquivos. Esses file URIs sao re-enviados ao Gemini em cada mensagem, mesmo quando o contexto ja foi absorvido. Nao ha nenhum mecanismo de caching server-side.

2. **Historico ilimitado (SYS-013):** O frontend (`chat-interface.tsx`, linhas 148-161) envia `messages.slice(startIdx)` como `history` — ou seja, TODAS as mensagens da conversa. Em conversas longas (20+ turnos), isso significa re-enviar dezenas de mensagens completas a cada request. Nao ha sliding window, truncation, nem estimativa de tokens. A API route (`route.ts`) repassa `history` diretamente para `streamMindChat()` sem nenhum controle de tamanho.

3. **Sessao recriada do zero (SYS-013):** Cada chamada a `streamMindChat()` e `createMindChat()` constroi o array de mensagens from scratch — nao ha reuso de sessao ou cache de contexto entre requests.

**Nota sobre Cached Content API do Gemini:** O Vercel AI SDK v6 (`@ai-sdk/google`) NAO suporta diretamente a Cached Content API do Google Gemini. Essa otimizacao exigiria uso direto do `@google/generative-ai` SDK com a API de caching, que e incompativel com o fluxo `streamText()` do Vercel AI SDK. Portanto, esta story foca em otimizacoes praticas e imediatas: history windowing, token estimation, e knowledge loading eficiente.

## Debitos Resolvidos
- **SYS-012**: Context window bloat — file URIs re-enviados em toda requisicao sem caching. Resolvido via knowledge caching em memoria e reducao de re-envio desnecessario.
- **SYS-013**: Stateless chat recreation — chat session recriada from scratch em cada `sendMessage`. Resolvido via history windowing com sliding window configuravel, token estimation, e resumo automatico opcional.

## Notas Tecnicas
- **Token estimation heuristica:** `Math.ceil(text.length / 4)` e uma aproximacao razoavel para texto em ingles/portugues. Para precisao real, seria necessario usar o tokenizer do modelo (ex: `tiktoken` ou equivalente Gemini), mas isso adiciona dependencia e complexidade desnecessaria nesta fase. A heuristica e suficiente para budget control.
- **File URIs nao contam no budget de historico:** Os file URIs do Gemini sao referencia server-side — o conteudo ja esta cacheado pelo Google. O custo de token e minimo (apenas os metadados). Portanto, NAO incluir file URIs no calculo de token do historico.
- **Sliding window vs summarization:** Esta story implementa sliding window (descartar mensagens antigas). Summarization automatica (resumir mensagens antigas em um paragrafo) e uma otimizacao futura (Fase 5) que exigiria uma chamada extra ao modelo para gerar o resumo.
- **CONTEXT_MAX_MESSAGES como safety net:** O limite de mensagens (50) e um fallback rapido para evitar payloads gigantes mesmo que a estimativa de token falhe. O limite de tokens (30K) e o controle primario.
- **Backward compatibility:** Se nenhuma env var for definida, os defaults sao aplicados silenciosamente. Zero breaking changes na API publica.
- **Cached Content API (futuro):** Quando o Vercel AI SDK v6 adicionar suporte a Gemini Cached Content API, sera possivel cachear o knowledge base server-side e evitar re-envio dos file URIs. Isso seria uma Story separada (Fase 5+). Alternativamente, um path hibrido usando `@google/generative-ai` diretamente para caching pode ser explorado.
- **Nao inclui:** Gemini Cached Content API (SDK nao suporta), summarization automatica de historico (futuro), per-mind context budgets (futuro), token counting real via tokenizer (futuro).
