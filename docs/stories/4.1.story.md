# Story 4.1: Setup Test Framework (Vitest + RTL + Playwright) + Unit Tests
**Epic:** 1 -- Transformacao Mentes Sinteticas
**Fase:** 4 -- Qualidade + Mobile + Acessibilidade
**Status:** InProgress
**Estimativa:** 8 SP (1 SP = ~2h)
**Criado por:** @sm (River)

## Descricao
Como desenvolvedor do Mentes Sinteticas, quero configurar um framework de testes completo (Vitest para unit tests, React Testing Library para component tests, Playwright para E2E) e escrever unit tests para todos os modulos utilitarios criticos — para que o projeto tenha cobertura de testes desde o inicio da Fase 4, regressions sejam detectadas automaticamente, e a qualidade do codigo seja mensuravel via metricas de coverage.

## Contexto
O projeto atualmente possui **zero testes** (debito SYS-017). Nenhum test runner esta configurado — `package.json` nao tem dependencias de teste nem scripts de teste. O `tsconfig.json` usa `moduleResolution: "bundler"` e `jsx: "react-jsx"`, o que e compativel com Vitest nativamente. O projeto usa path aliases (`@/*` -> `./src/*`) que precisam ser mapeados no config do Vitest. Todos os modulos utilitarios em `src/lib/` sao funcoes puras ou com dependencias minimas (Zod schemas, `process.env`), tornando-os candidatos ideais para unit tests.

Esta story foca em **setup + unit tests para modulos puros**. Component tests e E2E tests serao cobertos em stories subsequentes (4.2+).

## Debitos Resolvidos
- **SYS-017**: Zero test coverage — No unit, integration, or E2E tests. No test framework configured. (12-16h, P2). Parcialmente resolvido: setup completo + unit tests para modulos utilitarios. Component tests e E2E tests ficam para Story 4.2.

## Notas Tecnicas
- **Vitest vs Jest:** Vitest e preferido por suporte nativo a ESM, compatibilidade com Vite/React, e velocidade superior. Nao requer babel/ts-jest.
- **Singleton reset:** `getConfig()`, `getAIConfig()`, e `getContextConfig()` usam cache singleton (`_config`, `_aiConfig`, `_contextConfig`). Os testes precisam resetar esses singletons entre cada test case. Usar `vi.resetModules()` + re-import dinamico, ou exportar funcao `_resetConfigForTesting()` (prefixo underscore indica uso interno).
- **Environment mocking:** Usar `vi.stubEnv()` / `vi.unstubAllEnvs()` para mockar `process.env` nos testes de config. Alternativa: `beforeEach(() => { vi.stubEnv('KEY', 'value') })`.
- **Zod v4:** O projeto usa `zod@^4.3.6`. A API de `.safeParse()`, `.parse()`, `.transform()`, `.pipe()` e estavel. Testes devem usar `.safeParse()` para verificar tanto success quanto error cases.
- **Path aliases:** `vitest.config.ts` deve mapear `@/` -> `./src/` usando `resolve.alias` para que imports como `@/lib/errors` funcionem nos test files.
- **Coverage scope:** Coverage thresholds se aplicam APENAS aos modulos testados nesta story (`src/lib/utils.ts`, `src/lib/errors.ts`, `src/lib/config.ts`, `src/lib/ai/config.ts`, `src/lib/ai/context.ts`, `src/lib/validations/*.ts`). O coverage global do projeto sera menor (sem component/integration tests).
- **Playwright setup:** Apenas config nesta story. `npx playwright install` sera necessario para baixar browsers. Os E2E tests reais vem na Story 4.2.
- **Test file convention:** Testes colocados em `__tests__/` ao lado dos modulos (colocation pattern): `src/lib/__tests__/utils.test.ts`, `src/lib/ai/__tests__/config.test.ts`, etc.
- **Nao inclui:** Component tests (requer mocks de Next.js router, Supabase client), integration tests (requer DB), E2E tests (Story 4.2), mocking de Supabase/Drizzle.
