## Descricao
Como desenvolvedor do Mentes Sinteticas, quero escrever integration tests para as API routes (`/api/chat`, `/api/health`), services (`conversations`, `messages`, `minds`, `rate-limiter`), middleware de autenticacao, e E2E tests com Playwright cobrindo os fluxos criticos do usuario (login, navegacao, chat com mente) â€” para que a cobertura de testes va alem de unit tests de modulos puros e valide o comportamento integrado do sistema com mocks de Supabase/Drizzle e fluxos reais no browser.

## Contexto
A Story 4.1 configurou Vitest + RTL + Playwright e escreveu 111 unit tests para modulos puros (`utils`, `errors`, `config`, `ai/config`, `ai/context`, `validations/*`). Porem, as camadas mais criticas do sistema â€” API routes, services com DB, middleware de auth, e fluxos E2E â€” ainda nao possuem testes.

Os services (`conversations.ts`, `messages.ts`, `minds.ts`, `rate-limiter.ts`) dependem de Drizzle ORM + Supabase. Os API routes (`/api/chat`, `/api/health`) dependem de auth, rate limiting, e streaming. O middleware depende de `@supabase/ssr`. Todos requerem mocking adequado para testes de integracao.

Para E2E, o Playwright ja esta configurado (`playwright.config.ts`) com `baseURL: http://localhost:3000`, `webServer: npm run dev`, e 3 browsers (chromium/firefox/webkit). O diretorio `tests/e2e/` esta pronto com `.gitkeep`.

**Debitos resolvidos:**
- **SYS-017**: Zero test coverage â€” parcialmente resolvido em 4.1 (unit tests). Esta story completa com integration + E2E.

## Notas Tecnicas
- **Mocking strategy:** Integration tests usam mocks de Drizzle DB e Supabase Auth (nao conectam a DB real). O objetivo e testar a logica de integracao entre camadas (route -> service -> DB call), nao o DB engine em si. Testes contra DB real (se desejado) seriam uma story separada.
- **Drizzle mock pattern:** Mockar `@/db` exportando `db` como objeto com metodos chainable (`insert`, `select`, `update`, `delete`) que retornam mocks de `values`, `returning`, `from`, `where`, etc. Usar `vi.mock('@/db')`.
- **API route testing:** Testar as funcoes `GET`/`POST` exportadas diretamente, passando `Request` objects construidos manualmente. Nao requer servidor HTTP real.
- **Middleware testing:** Usar `NextRequest` e `NextResponse` do Next.js para simular requests. Mockar `createServerClient` do `@supabase/ssr`.
- **Streaming mock:** Para testar `/api/chat` POST success, mockar `streamMindChat` retornando um mock com `toTextStreamResponse()`. Nao e necessario testar streaming real no integration test (E2E cobre isso).
- **Playwright auth:** Usar `globalSetup` ou `setup` project para autenticar uma vez e reutilizar cookies/storage state em todos os E2E tests. Requer um usuario de teste existente no Supabase (criar via Dashboard ou seed script).
- **E2E environment:** E2E tests rodam contra `npm run dev` (dev server real). Requerem `.env.local` com credenciais Supabase validas e um usuario de teste. Considerar usar `.env.test` separado.
- **Test file convention:** Integration tests de services em `__tests__/` ao lado dos modulos (colocation). Integration tests de API routes em `__tests__/` dentro do diretorio da route. E2E tests em `tests/e2e/`.
- **Coverage:** Integration tests contribuem para coverage do Vitest. E2E tests (Playwright) nao contribuem â€” sao testes de sistema.
- **RLS simulation:** Nos integration tests, RLS e simulado via logica dos services (ex: `getConversationById` recebe `userId` e filtra). O teste verifica que a query inclui o filtro correto. Testes reais de RLS policies requerem conexao ao Supabase.
