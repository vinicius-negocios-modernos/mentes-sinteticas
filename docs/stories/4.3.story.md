# Story 4.3: CI/CD Pipeline (GitHub Actions + Vercel)
**Epic:** 1 -- Transformacao Mentes Sinteticas
**Fase:** 4 -- Qualidade + Mobile + Acessibilidade
**Status:** Done
**Estimativa:** 8 SP (1 SP = ~2h)
**Criado por:** @sm (River)

## Descricao
Como desenvolvedor do Mentes Sinteticas, quero configurar um pipeline de CI/CD com GitHub Actions para validacao automatica (lint, typecheck, unit tests, integration tests) em cada PR e push para `main`, e deploy automatico via Vercel (preview em PRs, production em `main`) — para que cada mudanca seja validada automaticamente antes do merge, reducindo regressoes e garantindo qualidade continua.

## Contexto
As Stories 4.1 e 4.2 estabeleceram uma suite robusta de testes: 111 unit tests, 75 integration tests via Vitest, e 15 E2E specs via Playwright. Porem, esses testes so rodam localmente — nao ha automacao de CI/CD. O deploy para Vercel e manual ou via integracao default sem quality gates.

O projeto usa Next.js 16.1.1 com Turbopack, TypeScript 5, ESLint, Vitest (unit + integration), e Playwright (E2E). O deploy target e Vercel. O banco de dados e Supabase (PostgreSQL) com Drizzle ORM.

Os E2E tests (Playwright) requerem um ambiente Supabase funcional e credenciais de usuario de teste, tornando-os mais adequados para execucao manual/opcional no CI inicialmente.

**Debitos resolvidos:**
- **SYS-016**: No CI/CD pipeline — resolvido completamente nesta story.

## Notas Tecnicas
- **Node.js version:** Usar Node 20 LTS no CI runner (`actions/setup-node@v4` com `node-version: '20'`).
- **Caching strategy:** Usar `actions/cache@v4` para cachear `node_modules` com key baseada em `package-lock.json` hash. Alternativa: usar `actions/setup-node` com `cache: 'npm'` (built-in caching).
- **Build env vars:** Next.js 16 valida `NEXT_PUBLIC_*` vars em build time. O CI precisa injetar pelo menos `NEXT_PUBLIC_SUPABASE_URL` e `NEXT_PUBLIC_SUPABASE_ANON_KEY` como secrets do repositorio para o build passar.
- **E2E no CI:** Playwright E2E requer Supabase real + usuario de teste. Por isso, o workflow E2E e manual (`workflow_dispatch`). Futuramente, pode-se usar Supabase local via Docker para E2E automatizado.
- **Vercel integration:** A integracao Vercel-GitHub e configurada pelo Vercel Dashboard (nao via workflow). O Vercel auto-detecta Next.js e configura build. Apenas env vars precisam ser adicionadas manualmente.
- **Branch protection:** Configurada via GitHub Settings > Branches > Branch protection rules. Nao e automatizavel via workflow, requer acesso admin ao repo.
- **Turbopack:** O build usa `next build` que automaticamente usa Turbopack em Next.js 16. Nao requer configuracao especial no CI.
- **Test parallelism:** Vitest roda testes em paralelo por default. No CI runner (2 cores), considerar `--maxWorkers=2` se houver problemas de memoria.
- **Playwright artifacts:** Em caso de falha E2E, o Playwright gera report HTML. O workflow faz upload como artifact para debugging.
