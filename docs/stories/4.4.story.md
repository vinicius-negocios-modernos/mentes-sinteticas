## Descricao
Como desenvolvedor do Mentes Sinteticas, quero integrar error tracking (Sentry), structured logging (substituindo console.log), e analytics (Vercel Analytics + Speed Insights) â€” para que erros em producao sejam capturados e rastreados automaticamente, logs sigam um formato estruturado com niveis e contexto, e metricas de performance (Web Vitals) sejam coletadas para monitorar a saude da aplicacao.

## Contexto
O projeto atualmente nao possui nenhuma forma de monitoring ou observability. Erros em producao passam despercebidos ate que um usuario reporte. O unico mecanismo de logging e `console.log` espalhado pelo codigo, sem niveis (info/warn/error), sem contexto estruturado, e sem persistencia. Nao ha metricas de performance (Web Vitals, API latency) sendo coletadas.

A Story 4.3 estabeleceu o pipeline CI/CD com GitHub Actions e Vercel, o que permite integrar source maps upload ao Sentry via CI e ativar Vercel Analytics no deploy.

O projeto usa Next.js 16.1.1 deployado na Vercel, com error boundaries ja implementadas (Story 3.4). A integracao com Sentry complementa os error boundaries existentes, capturando erros automaticamente e enviando-os para o dashboard.

**Debitos resolvidos:**
- **SYS-025**: No monitoring ou observability â€” resolvido completamente nesta story.
- **CROSS-005**: No observability pipeline â€” resolvido completamente nesta story.

## Notas Tecnicas
- **@sentry/nextjs:** SDK oficial do Sentry para Next.js. Configura automaticamente error handling para client, server, e edge runtime. O wizard (`npx @sentry/wizard@latest -i nextjs`) pode ser usado como referencia, mas a configuracao manual e preferida para controle total.
- **Source maps:** `@sentry/nextjs` plugin no `next.config.ts` faz upload automatico de source maps durante `next build`. Requer `SENTRY_AUTH_TOKEN` com scope `project:releases` e `org:read`.
- **Graceful degradation:** Toda integracao Sentry deve verificar se o SDK esta inicializado antes de chamar metodos. Quando `SENTRY_DSN` nao esta configurado, o SDK simplesmente nao inicializa â€” sem erros, sem side effects.
- **Logger design:** O logger deve ser simples e leve. Nao usar bibliotecas externas (winston, pino) â€” o output vai para stdout/stderr que Vercel captura automaticamente. Em producao, formato JSON. Em dev, formato legivel.
- **Vercel Analytics:** O componente `<Analytics />` de `@vercel/analytics` coleta page views e custom events. `<SpeedInsights />` de `@vercel/speed-insights` coleta Web Vitals (LCP, FID, CLS, TTFB). Ambos sao no-op fora da Vercel.
- **Performance tracing:** `tracesSampleRate` controla a porcentagem de transacoes capturadas. Em producao, 0.1 (10%) e suficiente para ter visibilidade sem exceder o free tier do Sentry. Em dev, 1.0 para debugging completo.
- **API instrumentation:** Usar `Sentry.startSpan()` para criar spans customizados no `/api/chat` route handler, medindo especificamente o tempo de resposta do Gemini SDK.
- **Testes:** Sentry deve ser mockado nos testes unitarios/integration (`vi.mock('@sentry/nextjs')`). O logger deve ter modo testavel ou ser mockavel.
- **Env vars novas:** `SENTRY_DSN`, `SENTRY_AUTH_TOKEN`, `SENTRY_ORG`, `SENTRY_PROJECT`, `SENTRY_TRACES_SAMPLE_RATE`. Todas opcionais (graceful degradation).
