# Story 4.5: Mobile-First Redesign (dvh, Touch Targets, Safe Areas)
**Epic:** 1 -- Transformacao Mentes Sinteticas
**Fase:** 4 -- Qualidade + Mobile + Acessibilidade
**Status:** Done
**Estimativa:** 4 SP (1 SP = ~2h)
**Criado por:** @sm (River)

## Descricao
Como usuario mobile do Mentes Sinteticas, quero que a interface funcione perfeitamente em dispositivos moveis — com viewport correto (sem barra do browser sobrepondo conteudo), areas de toque confortaveis, suporte a notch/safe areas, e layout responsivo adaptado — para que eu possa conversar com as mentes sinteticas de forma fluida em qualquer smartphone ou tablet.

## Contexto
O projeto atualmente apresenta problemas significativos de usabilidade mobile, identificados no debito tecnico UX-012 (classificado como ALTO pela @ux-design-expert):

**Problemas atuais:**
- **Viewport:** `h-[calc(100vh-140px)]` usado no `chat-interface.tsx` (linhas 300 e 314) ignora a barra de endereco do browser mobile, fazendo o conteudo ficar parcialmente escondido ou cortado. `100vh` nao desconta a UI do browser em Safari/Chrome mobile.
- **Touch targets:** Alguns elementos interativos tem area de toque menor que 44x44px (botao de delete na conversation-list, link "Encerrar Sessao" no header, botao "Sair" na nav).
- **Safe areas:** Nenhum `env(safe-area-inset-*)` e aplicado — em dispositivos com notch (iPhone X+), conteudo pode ficar atras do notch ou da home indicator bar.
- **Sidebar:** A conversation-list usa `hidden md:flex` (desaparece em mobile) sem oferecer alternativa mobile (drawer/sheet). Usuarios mobile nao tem acesso ao historico de conversas.
- **Layout:** A pagina home usa padding fixo `p-8 pb-20 sm:p-20` que desperdiça espaco em telas pequenas. O grid de mind cards e `grid-cols-1 md:grid-cols-2` (sem breakpoint para tablets maiores).
- **Inputs:** O textarea de chat pode causar zoom automatico no iOS se font-size < 16px.
- **Nav:** A nav de usuario (`layout.tsx`) usa `text-xs` e botoes pequenos, dificultando interacao em touch.

**O que ja existe (dependencias resolvidas):**
- Design System com shadcn/ui e tokens (Story 3.1)
- Componentes extraidos e organizados (Story 3.2) — facilita refactor isolado
- Chat UX redesenhado com textarea auto-resize, avatars, scroll (Story 3.5)
- Tailwind v4 configurado com suporte a classes responsivas

**Debito resolvido:**
- **UX-012**: Mobile viewport issues. `h-[calc(100vh-140px)]` ignora mobile browser chrome. Touch targets < 44px. Dependia de UX-002 (componentes extraidos, Story 3.2 — Done).

## Notas Tecnicas
- **`dvh` (Dynamic Viewport Height):** Unidade CSS que desconta a UI do browser mobile (barra de endereco, toolbar). Suportada em todos os browsers modernos (Chrome 108+, Safari 15.4+, Firefox 94+). Tailwind v4 nao tem classe nativa `h-dvh`, entao usar `h-[100dvh]` ou definir utility customizada em `globals.css`.
- **`viewport-fit=cover`:** Necessario no meta viewport para que `env(safe-area-inset-*)` funcione. Sem isso, o browser adiciona padding automatico que impede o app de usar a area do notch. Adicionar em `layout.tsx` via Next.js metadata viewport config.
- **`env(safe-area-inset-*)`:** CSS environment variables que retornam o tamanho do notch/home indicator. Valores: `safe-area-inset-top`, `safe-area-inset-bottom`, `safe-area-inset-left`, `safe-area-inset-right`. Usar com `max()` para fallback: `padding-bottom: max(env(safe-area-inset-bottom), 16px)`.
- **Touch targets 44x44px:** Apple HIG e WCAG 2.5.5 recomendam minimo 44x44px. Usar `min-h-11 min-w-11` (44px em Tailwind) ou padding que garanta a area total.
- **iOS auto-zoom:** Safari iOS faz zoom automatico em inputs com font-size < 16px. Solucao: garantir `text-base` (16px) ou maior em todos os `<input>` e `<textarea>`. Alternativa: `@supports` com `-webkit-text-size-adjust`.
- **shadcn/ui Sheet:** Componente de drawer/overlay. Instalacao: `npx shadcn@latest add sheet`. Suporta `side="left"` para sidebar mobile. Fecha com swipe ou click fora.
- **Tailwind v4 responsive:** Classes responsivas com prefixos `sm:`, `md:`, `lg:`, `xl:`. Mobile-first approach — classes sem prefixo sao o default (mobile). Adicionar prefixos para telas maiores.
- **Testes:** Componentes que recebem novas props (ex: ChatHeader com botao hamburger) precisam de testes atualizados. Verificar se mocks de `window.matchMedia` estao configurados nos testes existentes.
