# Story 4.6: Accessibility Pass 1 (ARIA, Live Regions, Focus, Contrast)
**Epic:** 1 -- Transformacao Mentes Sinteticas
**Fase:** 4 -- Qualidade + Mobile + Acessibilidade
**Status:** Done
**Estimativa:** 5 SP (1 SP = ~2h)
**Criado por:** @sm (River)

## Descricao
Como usuario com deficiencia visual, motora ou cognitiva do Mentes Sinteticas, quero que a interface seja acessivel — com ARIA attributes corretos, live regions para anuncios de novas mensagens, navegacao por teclado funcional, contraste de cores adequado, e indicadores de foco visiveis — para que eu possa interagir com as mentes sinteticas usando leitores de tela, teclado ou tecnologias assistivas.

## Contexto
O projeto atualmente apresenta falhas significativas de acessibilidade, identificadas no debito tecnico UX-005 (classificado como CRITICO pela @ux-design-expert):

**Problemas atuais:**
- **ARIA:** Zero atributos ARIA no projeto. Nenhum `aria-label`, `aria-live`, `aria-describedby`, ou `role` customizado. Leitores de tela nao conseguem interpretar a interface corretamente.
- **Live regions:** Container de mensagens do chat nao tem `aria-live="polite"`. Novas mensagens nao sao anunciadas para usuarios de leitores de tela.
- **Form labels:** Chat input e botao de enviar sem `aria-label`. Inputs de login/signup podem estar sem labels associados.
- **Keyboard navigation:** Tab order no chat nao verificado. Enter/Shift+Enter para enviar/quebrar linha funciona, mas nao ha indicacao visual de foco em muitos elementos.
- **Contraste:** Textos com `text-gray-500` e `text-gray-600` sobre fundo escuro falham no teste de contraste WCAG AA (ratio minimo 4.5:1 para texto normal, 3:1 para texto grande).
- **Focus indicators:** Sem estilos customizados de `focus-visible`. O outline padrao do browser e removido em muitos componentes sem substituto.
- **Skip link:** Nenhum link "Pular para conteudo" para usuarios de teclado.
- **Semantica HTML:** Hierarquia de headings nao verificada. Landmarks (`<nav>`, `<main>`, `<aside>`) podem estar ausentes ou incorretos.

**O que ja existe (dependencias resolvidas):**
- Design System com shadcn/ui e Radix UI primitives (Story 3.1) — Radix ja inclui muitos ARIA attributes automaticamente
- Componentes extraidos e organizados (Story 3.2) — facilita auditoria isolada
- Mobile-first com touch targets 44px (Story 4.5) — touch targets ja contribuem para WCAG 2.5.5

**Debito resolvido:**
- **UX-005**: Accessibility failures. Zero ARIA, zero live regions, zero form labels. Contrast failures. Dependia de UX-002 (componentes extraidos, Story 3.2 — Done).

**Escopo desta story (Pass 1 — fundacao):**
ARIA basics, live regions, form labels, keyboard nav, contraste, focus indicators, skip link, semantic HTML. Foco em WCAG 2.1 Level A com itens selecionados de Level AA.

**Adiado para Story 6.9 (Accessibility Pass 2):**
High contrast mode, reduced motion preferences (`prefers-reduced-motion`), focus trap no chat input durante loading, testes automatizados de a11y (axe-core), testes completos com NVDA/VoiceOver, auditoria completa WCAG AA.

## Notas Tecnicas
- **`aria-live="polite"`:** Anuncia mudancas ao leitor de tela quando o usuario esta ocioso (nao interrompe). Usar no container de mensagens para anunciar novas mensagens. `aria-relevant="additions"` garante que apenas novas adicoes sao anunciadas (nao remocoes).
- **`role="log"`:** Indica um container cujo conteudo cresce cronologicamente (ideal para chat). Leitores de tela tratam o conteudo de forma especial, anunciando novas entradas.
- **`role="status"`:** Para o indicador de streaming/digitando. Leitores de tela anunciam mudancas de status automaticamente.
- **Radix UI e ARIA:** Componentes shadcn/ui baseados em Radix ja incluem muitos ARIA attributes (Dialog, Sheet, DropdownMenu, etc.). Verificar antes de adicionar manualmente para evitar duplicacao.
- **Contraste WCAG AA:** Ratio minimo 4.5:1 para texto normal (< 18px), 3:1 para texto grande (>= 18px bold ou >= 24px). Ferramenta: WebAIM Contrast Checker. `text-gray-400` (#9ca3af) sobre `bg-gray-950` (#030712) tem ratio ~6.2:1 (passa). `text-gray-500` (#6b7280) sobre `bg-gray-950` tem ratio ~4.0:1 (falha).
- **`focus-visible` vs `focus`:** Usar `focus-visible` para mostrar indicadores apenas quando navegacao e por teclado (nao por mouse). Tailwind v4: `focus-visible:ring-2 focus-visible:ring-offset-2 focus-visible:ring-purple-500`. Definir globalmente para consistencia.
- **Skip link pattern:** `<a href="#main-content" class="sr-only focus:not-sr-only focus:absolute focus:top-4 focus:left-4 focus:z-50 ...">Pular para conteudo principal</a>` como primeiro filho do `<body>` (ou root layout).
- **Headings hierarchy:** Next.js pages devem ter um unico `<h1>` (titulo da pagina). Sub-secoes com `<h2>`, `<h3>` sequenciais. Evitar componentes que pulam niveis (ex: `<h1>` seguido de `<h3>`).
- **Testing ARIA:** Nos testes com Testing Library, usar `getByRole`, `getByLabelText`, `getByAltText` para verificar que ARIA attributes estao presentes. Exemplo: `screen.getByRole('log')` para o container de mensagens.
- **iOS VoiceOver:** `aria-live` funciona com VoiceOver no Safari iOS. Testar manualmente se possivel. `role="log"` suportado em todos os browsers modernos.
