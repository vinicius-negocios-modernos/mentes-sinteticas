# Story 4.8: Token Usage Tracking + API Cost Management
**Epic:** 1 -- Transformacao Mentes Sinteticas
**Fase:** 4 -- Qualidade + Mobile + Acessibilidade
**Status:** Done
**Estimativa:** 5 SP (1 SP = ~2h)
**Criado por:** @sm (River)

## Descricao
Como administrador/usuario do Mentes Sinteticas, quero que cada chamada a API de IA tenha seus tokens de entrada e saida rastreados, com custo estimado calculado e armazenado no banco de dados, e visibilidade na interface — para que eu possa monitorar o consumo de tokens, entender os custos reais da API Gemini, e configurar limites de uso diario/mensal que protejam contra gastos descontrolados.

## Contexto
O projeto atualmente nao possui nenhum rastreamento de tokens ou gestao de custos de API, identificado como debito tecnico CROSS-NEW-001 e Risk #5 do Epic 1 ("Custos de API nao controlados" — ALTA probabilidade, ALTO impacto).

**Problemas atuais:**
- **Zero tracking:** Nenhum registro de quantos tokens sao consumidos por conversa, por usuario, ou no total. Impossivel saber o custo real da operacao.
- **Sem visibilidade:** Usuario e administrador nao tem como saber quanto esta sendo gasto em API calls.
- **Sem limites de custo:** Rate limiting (Story 2.6) controla requests/minuto mas nao tokens consumidos. Um usuario pode fazer poucas requests mas consumir milhares de tokens cada.
- **Sem dados para otimizacao:** Sem metricas de token usage, impossivel identificar conversas ou mentes que consomem mais recursos.

**O que ja existe (dependencias resolvidas):**
- Rate limiter service: `src/lib/services/rate-limiter.ts` (Story 2.6) — controla requests, mas nao tokens
- Streaming chat API: `src/app/api/chat/route.ts` — usa Vercel AI SDK com `@ai-sdk/google`
- AI config com model settings: `src/lib/ai/config.ts` (Story 3.6) — presets balanced/creative/precise
- Context window optimization: `src/lib/ai/context.ts` (Story 3.7) — `truncateHistory`, token budget
- Database com Drizzle ORM: `src/lib/db/` (Story 2.1) — Supabase PostgreSQL
- Supabase com RLS policies em todas as tabelas existentes

**Debito resolvido:**
- **CROSS-NEW-001:** No token usage tracking or API cost management.

**Escopo desta story:**
Token counting via Vercel AI SDK usage data, schema de banco para armazenar uso, calculo de custo por modelo, indicador visual sutil na interface, e limites configuráveis de tokens diario/mensal.

## Notas Tecnicas
- **Vercel AI SDK usage data:** O `streamText` do Vercel AI SDK v4+ fornece dados de usage via callback `onFinish({ usage })` ou `result.usage`. O objeto `usage` contem `promptTokens` e `completionTokens`. Verificar a versao exata do SDK em uso e a API correspondente. Referencia: `@ai-sdk/google` provider.
- **Gemini pricing (referencia, verificar valores atuais):** gemini-2.0-flash: ~$0.10/1M input, ~$0.40/1M output. gemini-1.5-pro: ~$1.25/1M input, ~$5.00/1M output. Manter em arquivo de config separado para facil atualizacao.
- **RLS insert policy:** A rota `/api/chat` usa `createClient` server-side com service role para inserir na tabela `token_usage`. O SELECT e restrito ao proprio usuario via `auth.uid()`.
- **Performance:** A insercao na `token_usage` deve ser feita APOS o stream completar (no `onFinish`), nao bloqueando a resposta ao usuario. Considerar usar `Promise.allSettled` ou fire-and-forget para nao impactar latencia.
- **Budget check pre-request:** A verificacao de limites ANTES do request adiciona uma query ao banco. Para minimizar impacto, cachear o resultado por 1-5 minutos (ou usar o valor da ultima verificacao se recente). Alternativa: verificar apenas a cada N requests.
- **Numeric precision:** Usar `numeric(10,6)` no PostgreSQL para `cost_usd` — suficiente para valores muito pequenos (ex: $0.000150 por request).
- **Env vars:** `TOKEN_DAILY_LIMIT` (default: 500000), `TOKEN_MONTHLY_LIMIT` (default: 5000000). Adicionar ao `.env.example`.
- **UI indicator format:** Para tokens < 1000: "432 tokens". Para 1K-999K: "1.2K tokens". Para 1M+: "1.1M tokens". Custo: "$0.003" ou "$1.23".
- **Fallback graceful:** Se a insercao do token usage falhar (DB error), logar o erro mas NAO falhar a resposta do chat. O tracking e best-effort, nao deve impactar a experiencia do usuario.
- **Tabela existente `messages`:** Considerar adicionar `input_tokens` e `output_tokens` diretamente na tabela `messages` como alternativa simplificada. A tabela separada `token_usage` permite queries mais flexiveis e nao altera schema existente.
