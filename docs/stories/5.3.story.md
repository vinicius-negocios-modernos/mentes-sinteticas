**Epic:** 1 -- Transformacao Mentes Sinteticas

**Estimativa:** 3 SP (1 SP = ~2h)

## Descricao
Como desenvolvedor e mantenedor do Mentes Sinteticas, quero que o codebase esteja limpo, documentado e livre de problemas de higiene tecnica — para que novos contribuidores entendam rapidamente os modulos principais, segredos nao vazem pelo git, fontes renderizem de forma consistente, mensagens de erro sejam precisas, e logs de producao usem o logger estruturado em vez de console.log avulsos.

## Contexto
O projeto acumulou debitos tecnicos de higiene ao longo das Fases 1-4 que nao justificavam stories individuais mas que, agregados, impactam a manutenibilidade e a postura de seguranca do projeto.

**Problemas atuais:**
- **SYS-006 (.env tracked):** O arquivo `.env` foi rastreado no git em algum momento. Precisa garantir que `.gitignore` protege segredos e que `.env.example` esta completo e atualizado com todas as env vars do projeto.
- **SYS-020 (No code documentation):** Modulos criticos em `src/lib/ai/` e `src/lib/services/` nao possuem JSDoc nos exports. Dificulta onboarding e manutencao.
- **SYS-023 (Wrong error message):** Script de ingestao em `scripts/` exibe mensagem de erro que nao corresponde a condicao real de falha.
- **SYS-019 (Console.log in production):** Varios `console.log` espalhados no codigo de producao. O logger estruturado (`src/lib/logger.ts`) ja existe desde Story 4.4 mas nao foi adotado universalmente.
- **UX-011 (Competing font declarations):** Declaracoes de fonte duplicadas ou conflitantes entre `layout.tsx` e `globals.css`, causando inconsistencia na renderizacao tipografica.

**O que ja existe (dependencias resolvidas):**
- Logger estruturado: `src/lib/logger.ts` (Story 4.4) — pronto para substituir console.log
- `.env.example` existente (Story 1.3) — precisa verificacao de completude
- AI modules: `src/lib/ai/` com 10 arquivos exportando funcoes publicas
- Services: `src/lib/services/` com 5 arquivos de servicos
- Error handling: `src/lib/errors.ts` com hierarquia AppError

## Notas Tecnicas
- **Grep para env vars:** Usar `grep -rn 'process\.env\.' src/ scripts/` para listar todas as env vars referenciadas. Comparar com `.env.example` atual. Adicionar qualquer var faltante com placeholder e comentario.
- **Git history check:** `git log --all --diff-filter=A -- .env` para verificar se `.env` ja foi commitado. Se sim, considerar documentar que segredos podem ter vazado (nao e possivel remover do historico sem rewrite).
- **next/font pattern:** O padrao correto no Next.js 16 e usar `next/font/google` em `layout.tsx` e aplicar a classe CSS no `<body>`. Remover qualquer `@import url('fonts.googleapis.com/...')` de `globals.css` que duplique a mesma fonte.
- **Logger levels:** `logger.debug()` para info de desenvolvimento, `logger.info()` para operacoes normais (startup, requests), `logger.warn()` para situacoes recuperaveis, `logger.error()` para falhas. Manter a mesma estrutura de mensagem do console.log original.
- **JSDoc format padrao:**
  ```typescript
  /**
   * Brief description of what the function does.
   *
   * @param paramName - Description of the parameter
   * @returns Description of what is returned
   * @throws {AppError} When specific error condition occurs
   */
  ```
- **Console.log busca:** `grep -rn 'console\.log' src/` para listar todos os casos. Ignorar arquivos em `__tests__/`. Cada substituicao deve manter o mesmo conteudo informacional, apenas trocando o mecanismo de output.
- **README structure sugerida:** Project name + description, Tech stack, Prerequisites, Setup instructions (clone, install, env vars, DB), Development commands (dev, build, test, lint), Architecture overview (app structure, AI modules, services, DB layer), Deployment notes.
- **Impacto zero em runtime:** Todas as mudancas desta story sao: documentacao (JSDoc, README), configuracao (.gitignore, .env.example), ou substituicao de console.log por logger (mesma semantica). Nenhuma mudanca de logica de negocio.
