# Story 5.4: Docker Setup + Containerization
**Epic:** 1 -- Transformacao Mentes Sinteticas
**Fase:** 5 -- Polish & Launch Readiness
**Status:** Done
**Estimativa:** 3 SP (1 SP = ~2h)
**Criado por:** @sm (River)

## Descricao
Como desenvolvedor do Mentes Sinteticas, quero que o projeto tenha um setup Docker completo com Dockerfile multi-stage e docker-compose para desenvolvimento local — para que novos devs possam fazer onboarding em minutos com `docker compose up`, builds sejam reproduziveis em qualquer ambiente, e exista paridade entre desenvolvimento local e producao.

## Contexto
O projeto atualmente nao possui nenhuma forma de containerizacao, identificado como debito tecnico SYS-026 no technical-debt-assessment.md ("No Docker/containerization — Sem Dockerfile, sem docker-compose").

**Problemas atuais:**
- **Zero containerizacao:** Nenhum Dockerfile ou docker-compose existe no projeto. Cada dev precisa configurar o ambiente manualmente (Node.js, deps, env vars).
- **Onboarding lento:** Novo desenvolvedor precisa instalar Node.js na versao correta, rodar `npm install`, configurar `.env` manualmente — processo propenso a erros.
- **Sem paridade de ambiente:** Builds locais podem diferir entre maquinas (versoes de Node, OS, deps nativas). Nao ha garantia de que o que funciona local funciona em producao.
- **Builds nao reproduziveis:** Sem imagem Docker, deploys dependem inteiramente do Vercel build system. Nao ha como testar a build de producao localmente.

**O que ja existe (dependencias resolvidas):**
- Health check endpoint: `GET /api/health` (Story 2.7) — verifica DB + Auth
- CI/CD Pipeline: GitHub Actions + Vercel (Story 4.3) — build, lint, test automatizados
- `.env.example` com todas as env vars necessarias documentadas
- Next.js 16.1.1 com configuracao em `next.config.ts`
- Supabase e EXTERNO (cloud) — nao precisa ser containerizado

**Debito resolvido:**
- **SYS-026:** No Docker/containerization.

**Escopo desta story:**
Dockerfile multi-stage (deps, build, runner), .dockerignore, docker-compose.yml para dev local com hot reload, e documentacao de uso no README.md.

## Notas Tecnicas
- **Next.js standalone output:** Adicionar `output: 'standalone'` no `next.config.ts`. Isso faz o Next.js gerar um build self-contained em `.next/standalone/` que inclui apenas os arquivos necessarios para rodar em producao (sem `node_modules` completo). O entry point e `.next/standalone/server.js`. Referencia: documentacao oficial do Next.js em "Output File Tracing".
- **Multi-stage Dockerfile pattern:** Stage 1 (`deps`): instala dependencias com `npm ci --only=production` + copia `package.json`/`package-lock.json`. Stage 2 (`builder`): copia tudo, roda `npm run build`. Stage 3 (`runner`): copia APENAS `.next/standalone/`, `.next/static/`, e `public/` do builder. Resultado: imagem minima sem devDependencies e sem source code.
- **node:18-alpine:** Imagem base ~50MB vs ~350MB do node:18. Suficiente para Next.js standalone. Se houver problemas com deps nativas (sharp para otimizacao de imagens), adicionar `RUN apk add --no-cache libc6-compat`.
- **Supabase e EXTERNO:** Nao containerizar o banco. Supabase roda na nuvem. Apenas passar `NEXT_PUBLIC_SUPABASE_URL` e `NEXT_PUBLIC_SUPABASE_ANON_KEY` como env vars no container.
- **Hot reload em Docker:** Next.js usa `watchpack` para file watching. Em containers Docker, file system events do host podem nao propagar corretamente. Solucao: configurar `WATCHPACK_POLLING=true` no environment do compose para forcar polling-based watching. Impacto: levemente mais CPU, mas garante hot reload funcional.
- **Named volume para node_modules:** No docker-compose dev, montar `src/` do host no container para hot reload, mas usar named volume para `node_modules` para evitar que o `node_modules` do host (macOS) sobrescreva o do container (Linux alpine). Pattern: `- node_modules:/app/node_modules`.
- **Security no Dockerfile:** Criar usuario nao-root `nextjs` no stage runner: `RUN addgroup --system --gid 1001 nodejs && adduser --system --uid 1001 nextjs`. Rodar `USER nextjs` antes do CMD. Definir `ENV NODE_ENV=production`.
- **Porta e hostname:** Next.js standalone escuta em `0.0.0.0:3000` por default. Garantir `HOSTNAME=0.0.0.0` no environment do Dockerfile para que o container aceite conexoes externas.
- **Docker compose dev vs prod:** O compose e para DEV. Para producao, usar `docker build` + `docker run` direto (ou orquestrador). O compose usa `command: npm run dev` e monta volumes. A imagem de producao usa `CMD ["node", "server.js"]`.
- **Impacto no CI/CD:** A configuracao `output: 'standalone'` pode alterar o comportamento do build no Vercel. Verificar que o deploy no Vercel continua funcionando apos a mudanca. Vercel detecta automaticamente standalone output.
