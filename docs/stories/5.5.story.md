## Descricao
Como usuario do Mentes Sinteticas, quero que a aplicacao tenha feedback haptico sutil em interacoes-chave (envio de mensagem, selecao de mente) e infraestrutura de internacionalizacao preparada — para que a experiencia mobile seja mais imersiva e tactil, e o projeto esteja pronto para suportar multiplos idiomas no futuro sem refatoracao significativa.

## Contexto
Esta e a ultima story antes do lancamento. O foco e polish final: feedback haptico para melhorar a experiencia mobile e preparacao de i18n para escalabilidade futura.

**Problemas atuais:**
- **Sem feedback haptico (UX-NEW-007):** Interacoes no mobile nao tem nenhum feedback tactil. Enviar mensagem, selecionar mente — tudo acontece silenciosamente. Em apps modernos, feedback haptico sutil aumenta a percepcao de qualidade e responsividade.
- **Sem infraestrutura i18n (CROSS-004):** Todas as strings de UI estao hardcoded em portugues diretamente nos componentes. Adicionar um novo idioma exigiria editar dezenas de arquivos manualmente. Nao ha funcao de traducao, nao ha arquivo centralizado de mensagens.

**O que ja existe (dependencias resolvidas):**
- Mobile-first responsive design: Story 4.5 (touch targets 44px, dvh, safe areas, drawer)
- Acessibilidade: Story 4.6 (ARIA, live regions, focus-visible, contrast WCAG AA)
- Branding/SEO: Story 5.1 (favicon, meta tags, Open Graph — prerequisito para Lighthouse audit)
- Chat components: `src/components/chat/` (chat-interface, chat-message, chat-input, chat-header, chat-empty-state, conversation-list)
- Layout components: `src/components/layout/` (app-header, app-footer)
- Auth pages: `src/app/login/`, `src/app/signup/`
- Design system: shadcn/ui (New York), HSL tokens, dark default

**Debitos resolvidos:**
- **UX-NEW-007:** Sem feedback haptico/sonoro
- **CROSS-004:** No i18n infrastructure (prep only — full translation is OUT OF SCOPE)

**Escopo desta story:**
Utility de haptics com `navigator.vibrate`, integracao em chat-input e mind selection, respeitando `prefers-reduced-motion`. Extracao de strings para arquivo centralizado pt-BR, funcao `t(key)`, atualizacao dos componentes principais, e documentacao de como adicionar novo idioma. Lighthouse audit final para validar scores.

## Notas Tecnicas
- **Vibration API:** `navigator.vibrate(pattern)` aceita um numero (duracao em ms) ou array de numeros (vibrate, pause, vibrate...). Suporte: Chrome Android, Samsung Internet, Opera. NAO suportado: Safari (iOS/macOS), Firefox desktop. A funcao DEVE verificar `typeof navigator !== 'undefined' && 'vibrate' in navigator` antes de chamar.
- **prefers-reduced-motion:** Verificar via `window.matchMedia('(prefers-reduced-motion: reduce)').matches`. Idealmente, criar o MediaQueryList uma vez (module-level) e cachear o resultado para evitar re-queries. Considerar listener para mudancas dinamicas (`addEventListener('change')`).
- **Haptic patterns sugeridos:** `light` = `navigator.vibrate(10)` (tap sutil), `medium` = `navigator.vibrate(25)` (confirmacao), `confirm` = `navigator.vibrate([10, 50, 10])` (double-tap feedback). Manter patterns curtos — vibracao longa e irritante.
- **i18n architecture (prep-only):** A abordagem e intencional e simples: 1 arquivo TS por locale com objeto flat-ish, 1 funcao `t(key)` que faz lookup. NAO usar framework i18n completo (next-intl, react-i18next) nesta story — o objetivo e extrair strings e criar a infraestrutura minima. Framework completo pode ser adicionado depois se necessario.
- **Messages file structure sugerida:**
  ```typescript
  export const ptBR = {
    chat: {
      placeholder: 'Digite sua mensagem...',
      send: 'Enviar',
      emptyState: 'Inicie uma conversa...',
    },
    header: {
      title: 'Mentes Sinteticas',
    },
    auth: {
      loginTitle: 'Entrar',
      signupTitle: 'Criar conta',
      email: 'Email',
      password: 'Senha',
      // ...
    },
    footer: {
      copyright: '...',
    },
  } as const;
  ```
- **Funcao `t()` — dot notation:** Usar dot notation para keys (ex: `t('chat.placeholder')`). Implementar lookup recursivo no objeto de mensagens. Se key nao encontrada, retornar a propria key como fallback (facilita debug).
- **Type safety para `t()`:** Exportar tipo das keys validas usando template literal types do TypeScript. Ex: `type MessageKey = 'chat.placeholder' | 'chat.send' | ...`. Isso permite autocomplete e previne typos. Pode ser gerado automaticamente a partir do objeto de mensagens.
- **Lighthouse audit:** Rodar com `npx lighthouse http://localhost:3000 --output=json --output=html`. Ou usar Chrome DevTools > Lighthouse tab. Para CI, considerar `@lhci/cli` em story futura. Nesta story, audit manual e suficiente.
- **Visual consistency check:** Verificar: (1) todas as paginas usam tokens HSL do design system, (2) nenhum elemento com cor hardcoded fora do sistema, (3) tipografia consistente, (4) espacamento segue escala do Tailwind, (5) tema dark aplicado uniformemente.
- **Fallback graceful (haptics):** Se `navigator.vibrate` nao existe ou `prefers-reduced-motion` ativo, a funcao simplesmente retorna `false` sem efeito. Nenhum erro, nenhum log. Completamente silencioso.
- **SSR safety:** Tanto `navigator` quanto `window` nao existem no server-side (Next.js SSR/RSC). A funcao `triggerHaptic` e a verificacao de `prefers-reduced-motion` DEVEM ter guards `typeof window !== 'undefined'` ou ser usadas apenas em componentes client-side (`'use client'`).
