# Story 6.10: PWA / Offline Support
**Epic:** 1 -- Transformacao Mentes Sinteticas
**Fase:** 6 -- Features Legendarias
**Status:** InProgress
**Estimativa:** 5 SP (1 SP = ~2h)
**Criado por:** @sm (River)

## Descricao
Como usuario do Mentes Sinteticas, quero poder instalar o app no meu dispositivo (mobile ou desktop) e ter acesso a uma experiencia offline minima -- para que eu possa acessar o app rapidamente via home screen, carregar paginas mais rapido com cache de assets estaticos, e ler conversas recentes mesmo sem conexao a internet.

## Contexto
Atualmente o Mentes Sinteticas e um web app puro sem suporte a PWA. Isso significa:

1. **Sem instalacao:** O usuario nao recebe prompt de instalacao e nao pode adicionar o app a home screen com icone dedicado.
2. **Sem cache inteligente:** Cada visita recarrega todos os assets (JS, CSS, fontes, imagens) da rede, mesmo sem mudancas.
3. **Offline quebrado:** Sem conexao, o usuario ve a pagina de erro generica do browser em vez de uma experiencia branded.
4. **Sem leitura offline:** Conversas passadas sao inacessiveis sem rede, mesmo sendo conteudo ja visualizado.

**O que ja existe (dependencias resolvidas):**
- Web App Manifest em `src/app/manifest.ts` com name, icons, theme_color, background_color (Story 5.1)
- Icons: `src/app/icon.svg`, `src/app/apple-icon.tsx` (Story 5.1)
- Design system dark theme com glass-panel (`src/components/ui/`)
- Mobile-first responsive layout (Story 4.5)
- Static minds data em `src/data/minds.ts` (pode ser cacheado)
- Conversations service em `src/lib/services/conversations.ts` (server-side)
- Chat streaming via Vercel AI SDK (requer rede obrigatoriamente)
- Auth via Supabase (session em cookies)

**Escopo V1 desta story:**
- Service worker com cache de assets estaticos (cache-first)
- Pagina offline branded (fallback quando sem rede)
- App instalavel (manifest ja existe, SW registration completa o requisito)
- Cache de conversas recentes para leitura offline (read-only, via Cache API ou IndexedDB)

**Fora de escopo (V1):**
- Background sync (enviar mensagens enfileiradas ao reconectar)
- Push notifications
- Sincronizacao bidirecional de dados offline
- Cache de respostas de API genericas

## Notas Tecnicas
- **next-pwa choice:** `@ducanh2912/next-pwa` e o fork mantido ativamente para Next.js 13+/14+/15+. O pacote original `next-pwa` esta abandonado. Verificar compatibilidade com Next.js 16 antes de instalar -- se incompativel, usar Workbox diretamente via `next.config.ts` custom webpack config + `workbox-webpack-plugin`.
- **Workbox strategies:** CacheFirst para assets imutaveis (hashed filenames), NetworkFirst para HTML (conteudo pode mudar), NetworkOnly para API (dados sensiveis, autenticados). Referencia: https://developer.chrome.com/docs/workbox/modules/workbox-strategies
- **SW em development:** Desabilitar SW em dev (`disable: process.env.NODE_ENV === "development"`) para evitar cache stale durante desenvolvimento. Testar SW apenas no build de producao (`npm run build && npm start`).
- **IndexedDB para conversas:** Preferir `idb` (wrapper Promise-based) sobre API nativa por simplicidade. Schema: database `mentes-sinteticas-offline`, object store `conversations` com key `conversationId`. Indices: `cachedAt` (para FIFO cleanup).
- **iOS Safari limitacoes PWA:**
  - Service worker support limitado: SW pode ser terminado agressivamente pelo OS apos ~dias sem uso
  - Sem push notifications (ate iOS 16.4+ com limitacoes)
  - Sem background sync
  - Cache storage limitado (~50MB, pode ser purgado pelo OS sob pressao de armazenamento)
  - `standalone` mode nao suporta multi-window
  - Cookies/storage podem ser limpos apos 7 dias de inatividade (ITP)
  - Testar install flow via Safari "Add to Home Screen" -- nao ha install prompt automatico
- **Cache size management:** 200 entries e 30 dias para assets e conservador. Monitorar via `navigator.storage.estimate()` se disponivel. Respeitar `storage.quota` do browser.
- **Offline detection:** `navigator.onLine` e unreliable em alguns cenarios (pode retornar `true` mesmo sem internet real). Usar como heuristica, nao como fonte definitiva. Os eventos `online`/`offline` sao mais confiaveis para mudancas de estado.
- **Auth offline:** Supabase Auth session em cookies. Offline, o cookie pode existir mas nao pode ser renovado. Para leitura offline de conversas, nao validar auth server-side -- confiar no cache local (dados ja foram autorizados quando cacheados).
- **Manifest `id` field:** Campo `id` no manifest garante identidade unica da PWA entre instalacoes. Usar `"/"` como id. Sem isso, mudar `start_url` pode criar uma "nova" PWA em vez de atualizar a existente.
- **SW update flow:** Workbox `skipWaiting()` + `clientsClaim()` para ativar novo SW imediatamente. Considerar mostrar toast "Nova versao disponivel" com botao "Atualizar" (stretch, nao obrigatorio V1).
- **Pre-cache do offline page:** A pagina `/offline` DEVE ser incluida no precache manifest do SW para garantir disponibilidade quando totalmente offline. Configurar via `additionalManifestEntries` ou `runtimeCaching` com warmStrategy.
