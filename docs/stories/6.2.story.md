# Story 6.2: Multi-Mind Debates V1 — Round-Robin
**Epic:** 1 -- Transformacao Mentes Sinteticas
**Fase:** 6 -- Features Legendarias
**Status:** InProgress
**Estimativa:** 10 SP (1 SP = ~2h)
**Criado por:** @sm (River)

## Descricao
Como usuario do Mentes Sinteticas, quero selecionar 2 ou mais mentes e lancar um topico de debate para que elas discutam entre si em formato round-robin (turnos sequenciais) — para que eu possa observar perspectivas divergentes sobre um tema, aprender com o confronto de ideias, e intervir ou redirecionar a conversa quando desejar.

## Contexto
O Mentes Sinteticas atualmente suporta apenas conversas 1:1 entre o usuario e uma unica mente. O conceito de "debates multi-mente" e uma das features legendarias do produto — transformar o app de um chat individual em um "atheneum digital" onde mentes historicas debatem entre si.

**Arquitetura atual relevante:**
- **Chat API route:** `src/app/api/chat/route.ts` — POST endpoint com auth, rate limiting, token tracking, streaming via `streamMindChat()`
- **Streaming:** `src/lib/ai/stream.ts` — `streamMindChat()` usa `streamText` do Vercel AI SDK, constroi mensagens com knowledge injection via `buildStreamMessages()`
- **System prompts:** `src/lib/ai/prompts.ts` — `buildSystemPrompt(mindName)` gera prompt per-mind
- **Context:** `src/lib/ai/context.ts` — `truncateHistory()` com token budget (default 30K tokens, max 50 messages)
- **DB schema:** `conversations` (user_id, mind_id, title), `messages` (conversation_id, role, content, token_count)
- **Chat components:** `src/components/chat/` — chat-interface.tsx (client, manages state + streaming), chat-message.tsx (renders bubbles with avatars), chat-input.tsx, chat-header.tsx
- **Types:** `src/lib/types.ts` — `ChatMessage { role: "user" | "model"; text: string; timestamp?: Date }`
- **Minds:** `src/db/schema/minds.ts` — minds table with slug, name, title, systemPrompt, personalityTraits, greeting
- **Services:** `src/lib/services/minds.ts`, `conversations.ts`, `messages.ts`, `rate-limiter.ts`, `token-usage.ts`
- **Knowledge system:** per-mind knowledge docs injected via file URIs in `buildStreamMessages()`

**Desafios tecnicos:**
- Conversations table tem `mind_id` NOT NULL — debates envolvem multiplas mentes, requer novo tipo de conversa ou tabela separada
- Chat API route e 1:1 (mindName unico) — debates precisam de orchestracao server-side que chama multiplas mentes sequencialmente
- Token budget: cada turno de debate consome tokens para TODAS as mentes verem o historico completo — crescimento multiplicativo
- Streaming: cada resposta de mente precisa ser streamed individualmente ao cliente
- UI: precisa identificar visualmente qual mente esta "falando" — avatares, cores, nomes distintos

**Decisoes de design (V1 — escopo minimo):**
- Nova rota `/debate` com pagina dedicada (nao estender chat existente — complexidade separada)
- Nova tabela `debates` + `debate_participants` (nao poluir conversations com type field)
- Armazenar mensagens na tabela `messages` existente via conversation_id (debate cria uma conversation interna)
- Round-robin orchestrado no servidor: API route controla qual mente responde a seguir
- Usuario pode intervir a qualquer momento (insere mensagem no turno, round-robin continua apos)
- Max 4 mentes por debate (V1 — limitar complexidade de token)
- Max 5 rounds por debate (V1 — limitar custo)

### Schema DB — Debates
```sql
CREATE TABLE debates (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL,
  topic TEXT NOT NULL,
  max_rounds INTEGER NOT NULL DEFAULT 5,
  current_round INTEGER NOT NULL DEFAULT 0,
  status TEXT NOT NULL DEFAULT 'setup'
    CHECK (status IN ('setup', 'active', 'paused', 'completed')),
  conversation_id UUID REFERENCES conversations(id) ON DELETE SET NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE debate_participants (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  debate_id UUID NOT NULL REFERENCES debates(id) ON DELETE CASCADE,
  mind_id UUID NOT NULL REFERENCES minds(id),
  turn_order INTEGER NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(debate_id, mind_id),
  UNIQUE(debate_id, turn_order)
);
```

A tabela `debates` referencia `conversations` para reaproveitar a tabela `messages` existente para armazenar as mensagens do debate. Ao criar um debate, cria-se uma conversation "virtual" (com mind_id do primeiro participante como placeholder) e todas as mensagens do debate sao vinculadas a ela. O campo `role` em messages pode ser "user" (interjeccao) ou "assistant" (resposta de mente). Para identificar QUAL mente respondeu, o conteudo e prefixado com `[MIND:slug]` (removido na renderizacao) ou um campo adicional `metadata jsonb` pode ser adicionado a messages — avaliar tradeoff na implementacao.

**Decisao recomendada:** Adicionar coluna `mind_slug VARCHAR(255)` nullable na tabela `messages` (migration incremental). Mensagens de chat normal terao `mind_slug = NULL`, mensagens de debate terao o slug da mente que respondeu. Isso evita prefixos no conteudo e queries complexas.

### Logica Round-Robin
```typescript
function getNextParticipant(
  participants: DebateParticipant[],
  currentRound: number,
  totalTurnsCompleted: number
): DebateParticipant {
  const turnIndex = totalTurnsCompleted % participants.length;
  return participants.sort((a, b) => a.turnOrder - b.turnOrder)[turnIndex];
}
```

O `current_round` incrementa quando `totalTurnsCompleted % participants.length === 0` (todos falaram). O debate termina quando `current_round >= max_rounds`.

### Construcao de Contexto para Debate
Cada mente recebe:
1. **System prompt:** `buildSystemPrompt(mindName)` existente + instrucao de debate adicional
2. **Knowledge injection:** Mesmo fluxo de `buildStreamMessages()` — cada mente ve seus proprios docs
3. **Historico do debate:** Todas as mensagens anteriores (de todas as mentes + usuario) — truncadas com budget reduzido
4. **Mensagem atual:** Instrucao implicita "E a sua vez de responder no debate"

**Debate system prompt addon:**
```typescript
function buildDebatePromptAddon(topic: string, participants: string[], currentMind: string): string {
  const others = participants.filter(p => p !== currentMind);
  return `\n\nVoce esta participando de um debate sobre: "${topic}".
Outros participantes: ${others.join(', ')}.
Responda ao que foi dito pelos outros participantes. Defenda sua perspectiva.
Interaja diretamente com os argumentos apresentados. Seja respeitoso mas firme.
Mantenha sua personalidade e base suas respostas no seu conhecimento.
Resposta concisa — maximo 300 palavras.`;
}
```

### Streaming de Turnos
Cada turno e um streaming independente. O cliente:
1. Clica "Proximo Turno"
2. Recebe header `X-Current-Mind: [slug]` — mostra indicator de loading com nome da mente
3. Le o stream chunk a chunk (mesmo mecanismo de `chat-interface.tsx`)
4. Ao completar, adiciona mensagem ao estado local e habilita o botao "Proximo Turno" novamente

NAO e auto-play em V1 — o usuario controla o ritmo. Isso simplifica UX, evita rate limiting, e permite que o usuario leia cada resposta antes de avancar.

### Token Budget — Consideracoes
- Em um debate com 4 mentes e 5 rounds = 20 turnos de mente
- Cada turno envia o historico completo a mente atual
- Com respostas de ~300 palavras (~100 tokens) x 20 turnos = ~2000 tokens de historico no ultimo turno
- Knowledge injection adiciona ~2000-5000 tokens por mente
- Budget reduzido de 20K tokens para historico de debate e seguro para V1
- `maxOutputTokens: 1024` por resposta garante respostas concisas
- Token usage total por debate: estimativa ~40K-80K tokens (custo baixo com Gemini Flash)

### RLS (Supabase)

- `debates`: usuario so ve/edita seus proprios debates (`user_id = auth.uid()`)
- `debate_participants`: acesso via join com debates (usuario dono do debate)
- Avaliar se RLS ja e aplicado via Drizzle ou se precisa de migration SQL separada

**Risco migration:** A adicao de `mind_slug` na tabela `messages` e uma mudanca cross-cutting. Rodar testes existentes do chat apos migration para garantir que serialization e queries nao quebram.

### Dependencia de `conversations` para `messages`
O campo `conversations.mind_id` e NOT NULL. Para debates, criar a conversation com `mind_id` do primeiro participante como "host". Isso e um tradeoff aceitavel para V1 — evita alterar a constraint existente que quebraria o chat normal. Em V2, considerar tornar `mind_id` nullable ou adicionar `type` field.
