# Story 6.3: Mind Memory — Recall Cross-Session
**Epic:** 1 -- Transformacao Mentes Sinteticas
**Fase:** 6 -- Features Legendarias
**Status:** Draft
**Estimativa:** 7 SP (1 SP = ~2h)
**Criado por:** @sm (River)

## Descricao
Como usuario do Mentes Sinteticas, quero que cada mente "lembre" fatos importantes, preferencias e insights sobre mim entre conversas diferentes — para que a experiencia seja personalizada e evolua ao longo do tempo, criando um relacionamento mais profundo e autentico com cada mente.

Exemplo: ao iniciar uma nova conversa com Aristoteles, ele pode recordar que o usuario discutiu etica anteriormente, prefere dialogo socratico, e esta estudando filosofia — sem que o usuario precise repetir essas informacoes.

## Contexto
Atualmente, cada conversa comeca do zero. O sistema prompt (`src/lib/ai/prompts.ts`) injeta apenas a persona da mente e os knowledge documents. O historico de conversa (`src/lib/ai/context.ts`) e limitado a sessao atual via `truncateHistory`. Nao ha mecanismo para persistir ou recuperar informacoes sobre o usuario entre conversas.

**O que ja existe (dependencias resolvidas):**
- Supabase DB com Drizzle ORM e RLS em todas as tabelas (Fase 2)
- Conversations e messages persistidas no banco (Story 2.3)
- System prompt builder: `src/lib/ai/prompts.ts` (`buildSystemPrompt`)
- Context window com token budget: `src/lib/ai/context.ts` (`truncateHistory`, `estimateTokenCount`)
- Streaming via Vercel AI SDK: `src/lib/ai/stream.ts`, `src/app/api/chat/route.ts`
- Token usage tracking (Story 4.8)
- Auth + middleware protegendo /chat/*

**Problemas atuais:**
- Nenhuma informacao sobre o usuario persiste entre conversas
- Cada conversa e uma "pagina em branco" — a mente nao aprende nada sobre o usuario
- Nao ha mecanismo de extracao de fatos/preferencias
- Nao ha UI para o usuario gerenciar o que as mentes lembram sobre ele

**Escopo desta story:**
1. Nova tabela `mind_memories` com RLS policies
2. Extracao de memorias via AI ao final de cada conversa (post-stream)
3. Injecao de memorias relevantes no system prompt
4. API routes para CRUD de memorias (usuario gerencia seus dados)
5. UI de gerenciamento de memorias (visualizar e excluir)
6. Controle de token budget para memorias no contexto

## Notas Tecnicas
- **Nota:** Verificar tipo da coluna `minds.id` no schema existente (text slug vs uuid) e garantir que FK `mind_id` em `mind_memories` use o tipo correto.
- **Memory Extraction Prompt:** O prompt de extracao deve ser cuidadoso para nao extrair informacoes triviais. Foco em: (1) fatos pessoais declarados pelo usuario ("sou estudante de filosofia"), (2) preferencias de comunicacao ("prefiro exemplos praticos"), (3) topicos recorrentes de interesse, (4) insights que a mente percebe sobre o usuario. Ignorar: saudacoes, perguntas genericas, small talk.
- **Token Budget:** Memorias consomem espaco do context window. Default de 2000 tokens para memorias (env `MEMORY_MAX_TOKENS`). O chat route deve descontar esse valor do budget de historico: `maxHistoryTokens - memoryTokens`. Com ~500 chars por memoria e ~4 chars/token, cada memoria consome ~125 tokens, permitindo ~16 memorias no budget default.
- **Deduplicacao:** Normalizar `content` (lowercase, trim, remove pontuacao extra) antes de comparar. Usar query SQL com `lower(trim(content))` para check de existencia antes de INSERT.
- **Fire-and-Forget:** A extracao de memorias NAO deve bloquear a resposta do streaming. Executar dentro do `onFinish` callback com `.catch()` para capturar erros silenciosamente (log only). Similar ao pattern ja usado para `recordUsage`.
- **Privacy/GDPR:** O usuario tem controle total sobre suas memorias. Pode visualizar tudo que cada mente lembra e deletar individual ou em bulk. Nenhuma memoria e criada sem que o usuario tenha conversado com a mente. A API DELETE e a unica forma de remocao — nao ha soft delete.
- **RLS Design:** INSERT e UPDATE via service role only (backend) garante que apenas o servidor pode criar/atualizar memorias. SELECT e DELETE via `auth.uid() = user_id` garante que usuarios so acessam seus proprios dados.
- **Gemini para Extracao:** Usar o mesmo modelo configurado para chat (Gemini 2.0 Flash) para extracao. A extracao analisa apenas as mensagens da conversa atual, nao todo o historico. Structured output (JSON mode) para garantir formato consistente.
- **Env Vars novas:** `MEMORY_MAX_TOKENS` (default 2000) — limite de tokens de memorias no contexto. Adicionar ao `.env.example`.
- **Concorrencia:** Multiplas conversas simultaneas podem tentar extrair memorias. A deduplicacao por content normalizado previne duplicatas. Considerar um `ON CONFLICT DO NOTHING` no INSERT para safety.
- **Ordenacao:** Memorias mais recentes (`updated_at DESC`) tem prioridade no contexto. Memorias antigas que excedem o token budget sao simplesmente omitidas (nao deletadas).
- **Futuro (fora de escopo):** Relevance scoring (semantic search), memory summarization, memory decay (TTL), cross-mind memories.
